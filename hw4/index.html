Overview

In this homework, we built a cloth simulation system from scratch using a mass-spring model. Starting from constructing the grid of point masses and springs (Part 1), we moved to simulating the cloth's motion over time using Verlet numerical integration with damping and spring forces (Part 2). We then handled collisions with spheres and planes by projecting point masses back to valid positions upon penetration (Part 3). In Part 4, we handled self-collisions efficiently using a spatial hash so the cloth can fold onto itself without clipping through. Finally, in Part 5 we wrote a set of GLSL shaders covering diffuse, Blinn-Phong, texture mapping, bump and displacement mapping, environment-mapped reflections, and a custom toon shader.

One of the most interesting insights from this assignment was seeing how a few simple physical parameters (spring constant, density, damping) dramatically change the behavior of the cloth. Building intuition for these trade-offs through direct experimentation was the most valuable part of this assignment.

Part 2: Simulation via Numerical Integration

Effect of Spring Constant (ks)

Low ks: With a very low spring constant (e.g., ks = 2), the cloth is extremely soft and stretchy. It sags heavily under gravity, forms many small ripples during the fall, and takes longer to settle. The top edge droops significantly because the springs barely resist deformation.

High ks: With a very high spring constant (e.g., ks = 200000), the cloth behaves like a rigid sheet. It barely deforms, maintains a nearly flat shape throughout the simulation, and snaps into a near-flat resting position with minimal drape.

Effect of Density

Low density: With low density (e.g., 1 g/cm²), the cloth is very light. Gravity has less pull per unit area, so the cloth sags very little and the resting shape is nearly flat between the two pinned corners.

High density: With high density (e.g., 5000 g/cm²), the cloth is very heavy. The gravitational force per point mass is much larger, causing a pronounced sag in the middle and more extreme deformation during the fall.

Effect of Damping

Zero damping: With damping = 0, the cloth oscillates indefinitely. After falling it swings back and forth, never settling into a final resting state. The motion is chaotic and noisy.

Low damping (0.1%): The cloth swings a few times before slowly coming to rest. Oscillations are still visible but gradually decay.

Mid damping (0.4%): This is similar to the falling behavoir of a normal cloth, smoothly falls down and gradually comes to rest.

High damping (0.7~1.0%): The cloth falls very smoothly without any oscillation, as if moving through thick fluid. It settles directly into the final resting state with no swinging or bouncing.

Part 3: Handling Collisions with Other Objects

Implementation Description

Sphere: We first check whether the point mass is inside the sphere. If so, we project it to the sphere surface along the direction from the sphere center to the point mass, compute the correction vector from last_position to that surface point, and update the new position with friction scaling.

Plane: We first determine whether the point mass crosses the plane by checking the signs of its current and previous positions relative to the plane. If it crosses, we compute the line-plane intersection point, move it slightly back to the original side using SURFACE_OFFSET, and update the new position with friction scaling.

Collision integration: We update each unpinned point mass using Verlet integration with damping and acceleration. After storing the previous position in last_position, we test the updated point mass against every collision object. If a collision occurs, the collision object adjusts the point mass position accordingly.

Part 4: Handling Self-Collisions

Implementation Description

Without self-collision handling, the cloth passes straight through itself when it folds or stacks, which produces visibly broken results.

We implemented self-collision using a spatial hash so we do not need to compare every pair of point masses each frame. We partition space into 3D boxes with dimensions w = 3 * width / num_width_points, h = 3 * height / num_height_points, and t = max(w, h). For each point mass, we compute the integer box coordinates using floor(pos.x / w), floor(pos.y / h), and floor(pos.z / t), hash those coordinates into a single key, and insert the point mass into map[key]. This reduces the self-collision search to only the point masses that lie in the same spatial bucket.

In self_collide, we look up the current point mass's bucket, skip the point mass itself, and test only the candidate masses in that same cell. If another point mass is within 2 * thickness, we compute a correction vector pointing away from that neighbor so the two masses would be separated by exactly 2 * thickness. We sum all of these pairwise corrections, average them, divide by simulation_steps, and apply the final correction to the current point mass. Pinned masses are skipped so self-collision does not move fixed points. In simulate, we rebuild the spatial map every step and then call self_collide on each point mass after primitive collisions.

4.2: Effect of Density and ks

Changing the physical parameters noticeably changes how the cloth falls onto itself. A larger density gives each point mass more inertia, so the cloth drops faster, hits the plane harder, and tends to compress into a flatter and more compact pile before it settles. A smaller density makes the motion look lighter and the folds stay a bit more lifted.

To isolate the effect of spring stiffness, we kept density = 2 fixed and compared two values of ks. With a smaller spring constant (ks = 1000), the cloth behaves more softly: it bends more easily, collapses into deeper folds, and spreads into a looser stack. With a larger spring constant (ks = 5000), the cloth resists deformation more strongly, so the hanging part stays straighter and the folds are broader and less collapsed.

Part 5: Shaders

Shader Programs Overview

A shader program is a small GPU program that runs in parallel over many vertices or fragments. In our pipeline, the vertex shader is responsible for taking mesh attributes such as position, normal, tangent, and UV coordinates and transforming them into the coordinate system that the rest of the rendering code needs. The fragment shader then takes the interpolated results from the rasterizer and computes the final color of each visible fragment. In practice, the vertex shader sets up the geometric information, while the fragment shader decides how the surface should look under lighting, textures, reflections, and other material effects.

5.1: Diffuse Shading

For diffuse shading, we used the Lambertian term max(0, dot(n, l)) together with inverse-square light falloff. The vertex shader passes world-space positions and normals to the fragment shader, and the fragment shader computes how directly each point on the cloth faces the light. This creates a simple matte appearance that already makes the folds of the cloth readable.

5.2: Blinn-Phong Shading

The Blinn-Phong model adds two components on top of diffuse shading: an ambient term, which keeps the surface from going completely black in shadow, and a specular term, which produces highlights. Instead of reflecting the light vector directly, Blinn-Phong uses the halfway vector between the light direction and the view direction. When the normal aligns well with that halfway vector, we get a bright specular highlight. The full model is useful because each component contributes a different visual effect: ambient gives baseline visibility, diffuse gives shape, and specular gives shininess.

In the ambient-only render, the cloth is almost flat because every visible point receives nearly the same amount of light. In the diffuse-only render, the folds become much more legible because brightness depends on how the surface normal faces the light. In the specular-only render, most of the cloth is dark except for a few concentrated highlights where the viewing and lighting geometry aligns. The full Blinn-Phong result combines all three effects and produces the most convincing appearance.

5.3: Texture Mapping

For texture mapping, we sampled u_texture_1 using the interpolated UV coordinates and wrote the sampled color directly to the output. We replaced the default texture with our own image of the Campanile, so the cloth acts like a draped photo print rather than a uniformly shaded material. This makes it easy to see that the UV coordinates are being carried through the pipeline correctly.

5.4: Bump and Displacement Mapping

For both bump mapping and displacement mapping, we used the same stone-like height texture so the visual comparison would focus on the method rather than the input texture. In the bump shader, we sampled neighboring heights in texture space, estimated dU and dV, built a local normal (-dU, -dV, 1), and transformed it into world space using the TBN matrix. This changes the lighting response without changing the underlying geometry. As a result, the cloth appears to have embossed surface detail, but its silhouette stays essentially the same.

In displacement mapping, we moved each vertex along its normal by an amount proportional to the sampled height. Because the geometry itself changes, the effect is stronger than bump mapping and shows up in the silhouette and the large-scale shape of the draped surface. The displaced surface looks more physical, but it also depends much more on how many vertices are available to move.

The main visual difference is that bump mapping only changes shading, while displacement mapping changes actual geometry. Bump mapping is therefore cheaper and still effective for fine detail, but it cannot alter the outline of the object. Displacement mapping can create real geometric relief, but on coarse meshes it tends to look faceted or blocky because there are too few vertices to represent the height field smoothly.

This difference becomes especially clear when changing the sphere resolution from -o 16 -a 16 to -o 128 -a 128. Bump mapping changes very little with mesh coarseness because the effect is evaluated per fragment, so even a coarse sphere can still show detailed lighting variation. Displacement mapping, however, benefits directly from a denser mesh: the coarse sphere produces larger polygonal artifacts, while the finer sphere is able to represent smoother and more faithful geometric deformation.

5.5: Environment-Mapped Reflections

For the mirror shader, we computed the outgoing eye direction from the camera position and fragment position, reflected it across the surface normal, and sampled the cubemap in that reflected direction. This gives the cloth a polished reflective appearance without tracing any secondary rays. In the render below, the environment map is clearly visible as reflected sky and trees across the folds of the cloth.

Extra Credit: Custom Toon Shader

For the extra credit shader, we implemented toon (cel) shading in Custom.vert and Custom.frag. Instead of using a smooth diffuse term, we quantized dot(n, l) into a small number of intensity bands with a stepped function. We also used a thresholded specular term so the highlight appears as a sharp patch rather than a smooth glossy lobe. The result is a cartoon-like material with flat shaded regions and hard lighting transitions.

Compared with Blinn-Phong, the custom shader intentionally removes smooth variation and exaggerates the separation between light bands. On the sphere, this makes the shape read like a stylized animation render. On the cloth, the stepped lighting produces strong banding along the folds, which emphasizes the large drape structure and gives the scene a more graphic, non-photorealistic look.
